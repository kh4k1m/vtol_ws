# Руководство: Настройка визуальной навигации (Visual Odometry) для VTOL с использованием ROS 2 и ArduPilot SITL

В этом руководстве описан полный процесс настройки системы, при которой дрон (VTOL) использует камеру и метки AprilTag для определения своего положения в пространстве без использования GPS. Мы настроили симуляцию Hardware-In-The-Loop (HITL/SITL), где реальная камера и Jetson обрабатывают данные и отправляют их в виртуальный автопилот.

---

## 1. Архитектура системы

Наша система состоит из трех главных компонентов:
1. **Камера (IP-камера по RTSP)** — снимает видео и отдает поток по сети.
2. **Jetson (ROS 2)** — получает видео, распознает метку, вычисляет координаты и отправляет их по протоколу MAVLink.
3. **ArduPilot SITL (Симулятор)** — виртуальный полетный контроллер, который принимает эти координаты, обрабатывает их через фильтр Калмана (EKF3) и управляет виртуальным дроном.
4. **Mission Planner (Windows)** — наземная станция для визуализации полета и настройки параметров.

---

## 2. Настройка ROS 2 на Jetson

### 2.1. Получение видеопотока (`camera_node`)
Изначально нода камеры использовала утилиту `gphoto2`, которая делала 1 кадр в 3 секунды. Для навигации это слишком медленно.
Мы переписали `camera_node.cpp` на использование класса `cv::VideoCapture` из OpenCV. Теперь нода подключается к IP-камере по протоколу RTSP (`rtsp://192.168.1.110:8554/live`) и публикует кадры в топик `/camera/image_raw` с частотой ~30 FPS.

### 2.2. Распознавание меток (`tag_detector_node`)
Эта нода подписывается на изображения и использует библиотеку `cv::aruco` для поиска меток формата **AprilTag 36h11**. 
Когда метка найдена, алгоритм `solvePnP` вычисляет её 3D-положение (вектор трансляции и вращения) относительно камеры.

**Важный нюанс (Размер метки и калибровка):**
* Алгоритму нужно знать точный физический размер метки (мы указали `0.16 м`).
* Алгоритму нужно знать фокусное расстояние камеры (параметры `fx`, `fy`). Если они указаны неверно, дистанция по оси Z будет вычисляться с ошибкой. В идеале камеру нужно откалибровать по шахматной доске.

### 2.3. Мост MAVLink (`mavlink_bridge_node`)
Эта нода — связующее звено между ROS 2 и ArduPilot. Она берет координаты метки и отправляет их в автопилот.

**Ключевые изменения, которые мы внесли:**
1. **Преобразование систем координат:**
   Камера (OpenCV) и авиация (ArduPilot) используют разные системы координат. ArduPilot использует систему **NED** (North-East-Down: X-вперед, Y-вправо, Z-вниз).
   Поскольку наша камера смотрит *прямо вперед*, мы сделали следующее преобразование:
   * `X_ned = Z_camera` (глубина стала движением вперед)
   * `Y_ned = X_camera` (смещение вправо/влево)
   * `Z_ned = Y_camera` (смещение вверх/вниз)
2. **Отправка скорости:**
   Фильтр Калмана (EKF3) очень неохотно верит одним лишь координатам. Мы добавили ручное вычисление скорости (разница координат, деленная на время) и начали отправлять сообщение `VISION_SPEED_ESTIMATE`. Это резко повышает доверие автопилота к данным.
3. **Инициализация Origin:**
   Мы добавили автоматическую отправку сообщения `SET_GPS_GLOBAL_ORIGIN` (координаты 0,0,0) при старте ноды. Без этого EKF3 отказывался запускаться, выдавая ошибку `AHRS: waiting for home`.

## 3. Настройка симулятора ArduPilot (SITL)

Поскольку готовых сборок симулятора под архитектуру ARM64 (Jetson) нет, мы скомпилировали его прямо на Jetson из исходников с помощью системы сборки `waf`.

**Запуск симулятора:**
```bash
./build/sitl/bin/arduplane --model quadplane --home -35.363261,149.165230,584,353 --defaults Tools/autotest/default_params/quadplane.parm
```
Симулятор автоматически открывает TCP-порт `5760` для Mission Planner и `5762` для ROS 2.

---

## 4. Настройка параметров в Mission Planner (Non-GPS Navigation)

Чтобы автопилот начал использовать камеру вместо GPS, необходимо жестко перенастроить его "мозги" (фильтр EKF3). Это делается в `Config -> Full Parameter Tree`.

**Шаг 1: Отключение GPS**
* `GPS1_TYPE = 0` (Полностью отключаем виртуальный GPS симулятора, чтобы он не конфликтовал с камерой).

**Шаг 2: Указание источника данных для EKF3**
Мы говорим фильтру, чтобы он брал позицию и скорость из внешнего источника (ExternalNav), то есть из наших MAVLink-сообщений:
* `EK3_SRC1_POSXY = 6`
* `EK3_SRC1_POSZ = 6`
* `EK3_SRC1_VELXY = 6`
* `EK3_SRC1_VELZ = 6`

**Шаг 3: Настройка компаса (Yaw)**
Поскольку наша камера отправляет только позицию (мы принудительно обнулили Roll и Pitch в коде, чтобы не сбивать горизонт), автопилоту нужно откуда-то брать направление носа (Yaw).
* `EK3_SRC1_YAW = 1` (Использовать встроенный компас).

**Шаг 4: Включение модуля визуальной одометрии**
* `VISO_TYPE = 1` (Включает прослушивание MAVLink-сообщений VISION_POSITION_ESTIMATE).

**Шаг 5: Отключение проверок (для симулятора)**
* `ARMING_CHECK = 0` (Отключаем проверки перед взлетом).
* `EK3_GPS_CHECK = 0` (Разрешаем EKF3 стартовать без проверки качества GPS).

## 5. Порядок запуска всей системы

1. **Запуск симулятора на Jetson:**
   ```bash
   cd ~/ardupilot
   ./build/sitl/bin/arduplane --model quadplane --home -35.363261,149.165230,584,353 --defaults Tools/autotest/default_params/quadplane.parm
   ```
2. **Подключение Mission Planner:**
   На Windows-ПК открываем Mission Planner -> TCP -> IP Jetson (`192.168.0.79`) -> Порт `5760`.
3. **Запуск ROS 2 на Jetson:**
   ```bash
   cd ~/vtol_ws
   source install/setup.bash
   ros2 launch bringup system.launch.py
   ```
4. **Инициализация:**
   ROS 2 автоматически отправляет `EKF Origin`. Ошибка `waiting for home` исчезает.
5. **Навигация:**
   Показываем камере метку AprilTag. ROS 2 начинает слать координаты и скорости. В Mission Planner в окне EKF полоски `pos_horiz_abs` и `pos_vert_abs` становятся зелеными, и дрон на карте начинает двигаться вслед за меткой.

## Итог
Мы создали полноценный пайплайн визуальной навигации. Автопилот теперь "слеп" к спутникам, но "видит" мир через вашу камеру. В реальном полете этот же код будет работать точно так же, нужно будет лишь изменить порт подключения в `system.launch.py` с TCP на реальный Serial-порт (`/dev/ttyACM0`).
